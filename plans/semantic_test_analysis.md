# セマンティックテスト分析レポート

## 概要

セマンティックテストと他のテストの比較分析を行い、セマンティックテストの必要性を評価します。

---

## テストカテゴリ比較

### 1. セマンティックテスト (`semantic_assertions.rs`)

| 項目 | 詳細 |
|:---|:---|
| **テスト数** | 5（メインは`test_semantic_mass_verification`） |
| **カバレッジ** | 815能力、680カード |
| **パス率** | 87.1%（710/815） |
| **目的** | 日本語テキストから期待値を自動生成し、エンジン実行結果と比較 |

**特徴:**
- Python Oracleが日本語テキストを解析して期待値を生成
- ゾーンスナップショット（手札、デッキ、エネルギー等）の差分を検証
- 自動的にインタラクションを解決
- 大規模カードセットの一括検証が可能

### 2. オペコードテスト (`opcode_tests.rs`, `opcode_missing_tests.rs`, 等)

| 項目 | 詳細 |
|:---|:---|
| **テスト数** | ~30 |
| **カバレッジ** | 個別オペコード |
| **目的** | 特定オペコードの動作検証 |

**特徴:**
- 単一オペコードの入出力を検証
- エッジケースを明示的にテスト
- デバッグが容易

### 3. パリティテスト (`parity_tests.rs`)

| 項目 | 詳細 |
|:---|:---|
| **テスト数** | ~30 |
| **カバレッジ** | CPU-GPU間の同期 |
| **目的** | RustエンジンとWGSLシェーダーの一致確認 |

**特徴:**
- GPU実装との整合性確認
- ビットレベルの比較

### 4. Q&A検証テスト (`qa_verification_tests.rs`)

| 項目 | 詳細 |
|:---|:---|
| **テスト数** | ~20 |
| **カバレッジ** | 公式Q&Aルール |
| **目的** | 公式ルールへの準拠確認 |

**特徴:**
- 公式Q&Aのルールをテストケース化
- ユーザー報告のエッジケースをカバー

### 5. 再現テスト (`repro_*.rs`)

| 項目 | 詳細 |
|:---|:---|
| **テスト数** | ~30 |
| **カバレッジ** | 過去のバグ |
| **目的** | 回帰テスト |

**特徴:**
- 過去に発見されたバグの再現
- 修正が維持されていることを確認

### 6. 包括的テスト (`comprehensive_tests.rs`)

| 項目 | 詳細 |
|:---|:---|
| **テスト数** | ~25 |
| **カバレッジ** | パターン、トリガー、条件 |
| **目的** | 広範なパターン検証 |

**特徴:**
- トリガータイプ、条件、効果の網羅的テスト
- パターンベースの検証

## セマンティックテストの独自価値

### 他のテストではカバーできない領域

1. **大規模自動検証**
   - 815能力を一括検証できるのはセマンティックテストのみ
   - 新カード追加時に自動的にテスト対象に含まれる

2. **日本語テキストとの整合性**
   - カードテキスト（意図）とエンジン実装（実行）の乖離を検出
   - 翻訳ミスや解釈違いを発見

3. **累積デルタ検証**
   - 複数セグメントにわたる効果の累積を検証
   - 「その後」「さらに」等の接続詞で区切られた効果

4. **インタラクション自動解決**
   - 選択肢が必要な能力も自動的に解決して検証

### セマンティックテストの限界

1. **Oracle依存**
   - Python Oracleの解析精度に依存（現在78.3%→87.1%）
   - 正規表現の誤解析が6件発生

2. **合成環境**
   - 実際のゲームプレイとは異なる合成環境で検証
   - 条件が満たされないケースが102件

3. **デバッグ困難**
   - 失敗時の原因特定が難しい
   - 複数の要因が絡み合う

## 重複分析

### セマンティックテストと他のテストの重複

| 重複領域 | テスト | 重複度 | 判断 |
|:---|:---:|:---:|:---|
| 基本オペコード | `opcode_tests.rs` | 低 | **維持** - セマンティックは統合テスト、オペコードは単体テスト |
| ドロー効果 | `mechanics_tests.rs` | 中 | **維持** - セマンティックは実カード、mechanicsは汎用 |
| 条件チェック | `comprehensive_tests.rs` | 中 | **維持** - セマンティックは実データ、comprehensiveはパターン |
| Q&Aルール | `qa_verification_tests.rs` | 低 | **維持** - Q&Aは公式ルール、セマンティックはカードテキスト |

### 重複なしの領域

- **大規模カード検証**: セマンティックテストのみ
- **GPUパリティ**: `parity_tests.rs`のみ
- **回帰テスト**: `repro_*.rs`のみ

---

## 結論

### セマンティックテストは**必要**

**理由:**

1. **独自の価値**: 815能力の一括検証は他のテストでは不可能
2. **回帰防止**: 新カード追加時の自動検証
3. **テキスト整合性**: 日本語テキストとエンジンの乖離検出
4. **補完的役割**: 単体テスト（オペコード）と統合テスト（セマンティック）の二層構造

**改善推奨:**

1. **Oracle精度向上**: 正規表現の改良でパースミスを削減
2. **環境拡張**: 合成環境の条件充足率を向上
3. **失敗分析**: 失敗ケースのカテゴリ別対処

### テスト階層構造

```
┌─────────────────────────────────────────────┐
│           セマンティックテスト               │
│     (815能力、実カード、自動期待値生成)       │
├─────────────────────────────────────────────┤
│              包括的テスト                    │
│      (パターン、トリガー、条件網羅)          │
├─────────────────────────────────────────────┤
│    オペコードテスト  │  Q&A検証  │ 再現テスト │
│   (単体動作)       │ (公式ルール)│ (回帰)    │
├─────────────────────────────────────────────┤
│              パリティテスト                  │
│           (CPU-GPU同期)                      │
└─────────────────────────────────────────────┘
```

各層が異なる目的を持ち、相互に補完しているため、セマンティックテストを削除すると大規模検証能力が失われる。

| 優先度 | アクション | 効果 |
|:---:|:---|:---|

| 中 | 正規表現改良 | Oracleパースミス削減 |
